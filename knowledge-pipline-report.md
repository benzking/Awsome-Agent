## 一、项目概述

Knowledge Pipeline 是一个 Claude Code 技能（Skill），其核心理念是：

> **你的文档不该只是"被总结"。它们该被编译成一座可推理的知识体系。**

它将 PDF、图片、视频、Word、Excel、PPT、HTML 等多格式文档编译成结构化 Markdown 维基（Obsidian 兼容）+ 交互式知识图谱，支持跨文档矛盾检测、知识融合和深度推理链。

### 核心功能矩阵

| 功能 | 描述 |
|------|------|
| **/pipeline-ingest** | 多格式文档摄入，自动知识融合 |
| **/pipeline-query** | 多源聚合查询，展示共识与分歧 |
| **/pipeline-graph** | 生成交互式 vis.js 知识图谱 |
| **/pipeline-lint** | 检查维基完整性（断链、孤立页面、矛盾检测） |
| **/pipeline-ppt** | 基于知识库生成有深度的 LivePPT |
| **/pipeline-config** | 配置 LLM API |

## 二、核心特点

### 2.1 知识编译 vs 文档总结

传统工具做的是「总结」，Knowledge Pipeline 做的是「编译」：

- **总结**：把 20 份文档压缩成一堆泛泛的要点
- **编译**：将文档编译成实体网络、概念图谱、矛盾检测结果

关键区别在于中间表示（Intermediate Representation）：

```
传统工具：文档 → 黑盒向量数据库（不可读）
KP：文档 → Markdown 维基（可浏览、可编辑、可版本控制）
```

### 2.2 增量知识积累

- 普通工具：每次会话重新上传，覆盖旧内容
- KP：新文档自动合并到已有实体/概念页面，**知识只增不减**

### 2.3 主动矛盾检测

每次摄入新文档后自动扫描，与已有 claims 对比，主动报告：

```
⚠️ 矛盾发现：
- [paper-A] "GPT-4 在代码生成任务上准确率 92%"
- [paper-B] "GPT-4 代码生成准确率仅 78%"
差异可能源于评估基准不同
```

### 2.4 深度推理链（BFS 路径搜索）

查询时加上 `--rc` 标志，基于知识图谱进行 BFS 路径搜索，展示节点之间的推理路径：

```
不对称相似度 ←→ 群组融合 ←→ 学术编码 ←→ AI 替代效应
```

### 2.5 本地运行 + 多 LLM 支持

- 完全本地运行，数据不出机
- 支持任意 OpenAI 兼容 API：OpenAI / DeepSeek / 火山引擎 / Together AI / Ollama（本地）

## 三、与其他热门项目的对比

### 3.1 vs ChatGPT/Claude 文件上传

| 维度 | ChatGPT/Claude | Knowledge Pipeline |
|------|---------------|-------------------|
| 知识积累 | 每次会话重新上传，不积累 | 一次摄入，永久积累 |
| 跨文档关联 | 只检索相关片段拼接 | 自动构建实体和概念网络 |
| 矛盾检测 | 不会主动发现矛盾 | 摄入时立即报告冲突 |
| 知识融合 | 不存在，每次独立 | 新文档自动合并到已有页面 |
| 多源视角 | 只给一个综合答案 | 展示每个源的立场 + 共识与分歧 |
| 中间表示 | 黑盒，用户不可见 | Markdown 维基，可浏览可编辑 |

### 3.2 vs NotebookLM

| 维度 | NotebookLM | Knowledge Pipeline |
|------|-----------|-------------------|
| 运行位置 | Google 云端，数据上传到 Google | 本地运行，数据不出机 |
| LLM 选择 | 仅 Google Gemini | 任意 OpenAI 兼容 API / 本地 Ollama |
| 矛盾检测 | 无 | 主动跨源矛盾检测 |
| 知识融合 | 无，源文档独立 | 自动实体/概念合并 |
| 知识图谱 | 无 | 交互式 vis.js 图谱 + 社区检测 |
| 推理链 | 无 | BFS 深度推理链 + 可视化 |
| 可导出性 | 锁定在 Google 生态 | 纯 Markdown 文件，Obsidian/Git 兼容 |
| 格式支持 | PDF、文本、网页 | PDF/图片/视频/Word/Excel/PPT/HTML |

### 3.3 vs 传统 RAG

| 维度 | 传统 RAG | Knowledge Pipeline |
|------|---------|-------------------|
| 工作原理 | 切片 → 向量化 → 检索相似片段 | 编译 → 结构化知识网络 → 推理 |
| 回答依据 | 文本碎片相似度匹配 | 概念网络中的关系推理 |
| 跨文档能力 | 弱 — 只是把更多碎片扔进上下文 | 强 — 实体/概念页面跨源自动合并 |
| 矛盾处理 | 不感知，可能给出矛盾答案 | 主动检测并报告 |
| 中间表示 | 向量数据库（不可读） | Markdown 维基（人类可读可编辑） |
| 开发成本 | 需要写代码搭 pipeline | 5 个斜杠命令，零代码 |
| 新文档适配 | 需要重新索引 | 增量摄入，自动融合 |

### 3.4 vs AI PPT 工具（Gemini/GPT/Gamma）

| 维度 | Gemini/GPT/Gamma | Knowledge Pipeline LivePPT |
|------|-----------------|---------------------------|
| 内容来源 | 用户输入的一句提示词 | 知识库：多份文档 → 实体网络 → 概念图谱 |
| 分析深度 | 模型通用知识，泛泛而谈 | 你自己文档中的真实数据和洞察 |
| 矛盾处理 | 不感知，可能自相矛盾 | 自动检测跨源矛盾，标注分歧 |
| 来源追溯 | 无来源标注 | 每页标注 Wiki 来源 |
| 编辑方式 | 重新生成整副幻灯片 | 自然语言逐页实时编辑，不破坏其他页 |
| 导出格式 | PDF/图片（不可编辑） | PPTX（可编辑文本框 + 高清背景图） |
| 模板 | 平台固定模板 | 8 种内置主题，一键选择 |
| 成本 | $20–40/月订阅 | 免费开源，用自己的 LLM |

## 四、优缺点分析

### ✅ 优点

1. **本地运行，数据隐私有保障**：不依赖云端，数据不出机
2. **增量积累，知识不丢失**：一次摄入永久累积，新文档自动融合
3. **主动矛盾检测**：摄入时主动发现跨源冲突，而非被动回答
4. **中间表示人类可读**：Markdown 维基可浏览、可编辑、可版本控制
5. **支持多格式摄入**：PDF/图片/视频/Word/Excel/PPT/HTML 等
6. **零代码使用**：5 个斜杠命令即可完成所有操作
7. **推理链可视化**：BFS 路径搜索 + reasoning.html 交互式展示
8. **LivePPT 有深度**：基于知识库生成有引用来源的分析报告

### ❌ 缺点

1. **依赖 Claude Code**：主要设计为 Claude Code 技能，非独立工具
2. **没有官方 API**：难以集成到现有系统
3. **Git 克隆不稳定**：从描述中可见，当前仓库 clone 存在 EOF 问题（可能网络或仓库问题）
4. **图谱依赖 vis.js**：需要在浏览器打开，静态展示能力弱
5. **PPT 生成质量依赖 LLM**：如果 LLM 理解能力不足，生成内容可能不够精准
6. **缺乏团队协作机制**：面向个人用户，无多用户权限管理
7. **中文文档支持待验证**：README 有中英文，但实际使用中文化支持可能不足

## 五、定位与适用场景

### 一句话总结

| 工具 | 本质 |
|------|------|
| ChatGPT/Claude | 对话工具，每次从零开始 |
| NotebookLM | 文档阅读器，单项目单次使用，锁定 Google |
| 传统 RAG | 搜索引擎，在碎片中查找 |
| Knowledge Pipeline | **知识编译器**，把文档编译成可推理的知识体系 |

### 适用场景

- **学术研究**：读 30 篇论文后，想知道"A 和 B 论文对某问题有没有矛盾"
- **尽职调查**：分析 5 份审计报告的安全策略异同
- **竞品分析**：对比 OpenAI/Anthropic/Google 在 AI 安全上的策略分歧
- **知识管理**：不想用黑盒向量数据库，需要能看到知识怎么组织

### 不适用场景

- **快速简单问答**：直接问 ChatGPT 更快
- **团队协作**：缺乏多用户和权限管理
- **非结构化数据探索**：没有内置的统计或可视化分析

## 七、总结

Knowledge Pipeline 的核心创新在于**从「检索」到「推理」的范式转变**：

- 传统 RAG：在文档碎片中搜索答案
- KP：在知识体系中推理答案

它解决了一个根本问题：**文档被摄入后不是消失了，而是成为了一座可推理的知识网络**。增量积累 + 主动矛盾检测 + 推理链可视化，这是现有工具链中罕见的组合。

但它本质上是**个人知识管理工具**，而非企业级解决方案。依赖 Claude Code、缺乏 API、团队协作弱，都是在向更广泛场景扩展的障碍。
