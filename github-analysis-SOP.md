# GitHub 项目分析 SOP · 标准作业流程

> 版本：v1.1
> 创建日期：2026-04-28
> 更新日期：2026-04-28（新增报告审核环节）
> 适用场景：分析任意 GitHub 项目，输出标准化报告并存入知识库

---

## 一、流程总览

```
① 接收任务        →  用户发送 GitHub 项目链接
       ↓
② 获取项目        →  克隆或 Web 获取 README
       ↓
③ 深度阅读        →  抓取项目文档 + 关键源码
       ↓
④ 撰写报告初稿    →  按模版输出 markdown 报告
       ↓
⑤ 报告审核        →  3 个子 Agent 多角度评审 + 1 个 Agent 最终审稿
       ↓
⑥ 文件组织        →  按目录结构存放
       ↓
⑦ 更新索引        →  README 作为项目索引
       ↓
⑧ 提交推送        →  Commit + Push 到 GitHub
```

---

## 二、详细步骤

### Step 1：接收任务

用户发送 GitHub 项目链接，格式如：
```
https://github.com/{owner}/{repo}
```

**提取关键信息：**
- `owner` = 项目所有者
- `repo` = 仓库名称
- 规范 slug 格式：`{repo}` 转小写，多个单词用 `-` 连接

---

### Step 2：获取项目

#### 方式 A：Git Clone（推荐用于完整分析）

```bash
# 清理旧目录（如果存在）
rm -rf /path/to/{repo-slug}
# 浅克隆（--depth 1）避免网络超时
git clone --depth 1 https://github.com/{owner}/{repo}.git
```

#### 方式 B：Web Fetch（克隆失败时备选）

```bash
# 抓取主页（英文版优先）
web_fetch(url="https://github.com/{owner}/{repo}")
web_fetch(url="https://github.com/{owner}/{repo}/blob/main/README_EN.md")
```

---

### Step 3：深度阅读

**抓取优先级（按顺序）：**

| 优先级 | 文件 | 原因 |
|--------|------|------|
| ⭐⭐⭐ | `README.md` / `README_EN.md` | 项目概述 |
| ⭐⭐ | `SKILL.md` / `CLAUDE.md` / `agent.py` / `main.py` | 入口逻辑 |
| ⭐⭐ | `package.json` / `setup.py` / `requirements.txt` | 依赖环境 |
| ⭐ | `examples/` 目录 | 快速上手 |

**若克隆失败：** 用 `web_fetch` 获取 README 内容，结合 GitHub 页面标题和描述补充信息。

---

### Step 4：撰写报告初稿

**输出路径：**
```
/home/gly/.openclaw/workspace/Awsome-Agent/llm-wiki/{repo-slug}/report.md
```

**报告结构（按模版 v1.1）：**

```
一、精要介绍（200字以内）
二、项目概览（Stars / Forks / 语言 / 协议 / 标签）
三、应用标签分类
四、核心特点（3-5 个要点）
五、工作流程（Mermaid 流程图或有序列表）
六、应用场景（3-5 个典型场景）
七、对比同类热门项目（横向对比表格）
八、安装指引（环境要求 + 步骤 + 快速验证）
九、项目结构（目录树）
十、关键文件清单
十一、网上实践案例（≤5 个，精选深度分析）
```

---

### Step 5：报告审核（关键新增环节）

> ⚠️ 初稿完成后必须执行此环节，不得跳过

#### 5.1 评审 Agent 配置

**Agent 1：内容正确性审核**
- 角色：技术评审
- 关注点：事实是否准确、描述是否客观、数据是否最新
- 输出：「正确性审核报告」

**Agent 2：内容完整性审核**
- 角色：结构评审
- 关注点：模版章节是否完整、对比表格是否充分、案例是否足够
- 输出：「完整性审核报告」

**Agent 3：实用价值审核**
- 角色：用户视角评审
- 关注点：安装指引是否可落地、应用场景是否真实、案例是否可借鉴
- 输出：「实用价值审核报告」

**Agent 4：最终审稿（整合修订）**
- 角色：主编
- 任务：综合三份审核报告，修订报告内容，输出最终版本
- 输出：「最终审稿报告」

#### 5.2 评审执行方式

使用 `sessions_spawn` 启动隔离子 Agent：

```
sessions_spawn(
  runtime="subagent",
  context="fork",      # 需要当前报告内容作为上下文
  task="你是技术评审，请阅读报告初稿并输出『正确性审核报告』..."
)

sessions_spawn(
  runtime="subagent",
  context="fork",
  task="你是结构评审，请阅读报告初稿并输出『完整性审核报告』..."
)

sessions_spawn(
  runtime="subagent",
  context="fork",
  task="你是用户视角评审，请阅读报告初稿并输出『实用价值审核报告』..."
)
```

三份审核报告收集完毕后，调用最终审稿 Agent：

```
sessions_spawn(
  runtime="subagent",
  context="fork",
  task="你是主编，综合三份审核报告，对报告初稿进行修订，输出『最终审稿报告』..."
)
```

#### 5.3 审核维度说明

| Agent | 审核维度 | 检查要点 |
|-------|---------|---------|
| Agent 1 技术评审 | 正确性 | README 数据是否准确、版本号/Stars/Forks 是否正确、功能描述是否与源码一致 |
| Agent 2 结构评审 | 完整性 | 模版 11 个章节是否齐全、对比表格竞品数量是否 ≥3、安装步骤是否完整 |
| Agent 3 用户视角 | 实用价值 | 安装命令是否可执行、案例是否真实可落地、应用场景是否具体 |
| Agent 4 主编 | 整合修订 | 综合三份报告修订内容，确保逻辑连贯、无矛盾、表述清晰 |

### Step 6：文件组织规范

**目录结构：**

```
Awsome-Agent/
├── README.md              # 项目索引（所有分析报告的入口）
├── github-analysis-template.md  # 模版文件（固定存放）
├── github-analysis-SOP.md       # SOP（固定存放）
└── llm-wiki/              # 知识库根目录
    └── {repo-slug}/       # 每个项目一个文件夹
        └── report.md      # 分析报告（固定命名）
```

**命名规范：**
- 文件夹：`{repo-slug}` 小写（例：`knowledge-pipline`）
- 报告：`report.md`（固定，不可自定义）
- 模版：`github-analysis-template.md`（固定）
- SOP：`github-analysis-SOP.md`（固定）

---

### Step 7：更新 README 索引

每次新增报告，必须同步更新仓库根目录的 `README.md`：

```markdown
## 📚 分析报告

### {项目名}

**{一句话介绍核心功能}** — {简要特点}。

👉 [查看完整分析报告](./llm-wiki/{repo-slug}/report.md)
```

---

### Step 8：提交并推送

```bash
cd /home/gly/.openclaw/workspace/Awsome-Agent

# 添加所有变更
git add -A

# 提交（规范 commit message）
git commit -m "docs: add {repo-slug} analysis report"

# 推送（处理网络抖动）
GIT_TERMINAL_PROMPT=0 git push -u origin main
```

**Commit Message 规范：**
- `docs: add {repo-slug} analysis report` — 新增报告
- `docs: update {repo-slug} report` — 更新报告
- `docs: restructure` — 目录重组

## 三、格式要求汇总

| 内容 | 格式要求 |
|------|---------|
| 精要介绍 | ≤200字，含项目定位、核心功能、目标用户 |
| 项目概览 | 表格形式，字段固定（地址/协议/Stars/Forks/语言/标签） |
| 应用标签 | 从 Agent / Plugin / Skill / MCP / Tool / 应用 等选取 |
| 核心特点 | 3-5 个要点，每点 ≤1 行 |
| 工作流程 | Mermaid 或有序列表，不超过 10 步 |
| 应用场景 | 3-5 个，每个场景一句话描述 |
| 对比表格 | 3-5 个竞品，维度包括定位/能力/运行方式/存储/LLM支持/安装成本 |
| 安装指引 | 环境要求 → 安装步骤 → 快速验证 |
| 网上案例 | ≤5 个，每个案例含来源/场景/做法/效果/借鉴点 |
| Commit Message | `类型: 简短描述`，类型用 docs/feat/fix 等 |

## 四、关键教训（从实操中提炼）

> ⚠️ 以下问题曾在操作中遇到，需特别注意：

1. **克隆超时/EOF**
   - 原因：网络不稳定、仓库过大
   - 解决：使用 `--depth 1` 浅克隆，或改用 `web_fetch`

2. **文件已存在**
   - 原因：重复克隆到同一路径
   - 解决：先 `rm -rf {dir}` 再克隆

3. **未推送到远程就结束**
   - 要求：每次完成操作后，必须 `git push` 到 GitHub

4. **README 未同步更新**
   - 要求：新增报告后，必须同步更新 README 索引

5. **跳过审核环节**
   - 要求：Step 5 报告审核是必做环节，三轮评审 + 最终审稿缺一不可

6. **目录结构混乱**
   - 解决：严格按 `llm-wiki/{repo-slug}/report.md` 存放

## 五、检查清单

每次分析完成后，逐项核对：

```
☐ GitHub 仓库已获取（clone 或 web_fetch）
☐ README 已阅读（英文优先）
☐ 报告初稿已撰写（模版 11 个章节）
☐ 3 个子 Agent 已完成多角度评审
☐ 最终审稿 Agent 已完成整合修订
☐ 最终报告已写入 llm-wiki/{repo-slug}/report.md
☐ README.md 索引已更新（含一句话介绍 + 链接）
☐ git add + commit 已执行
☐ git push 已成功
```

## 六、子 Agent 评审 Prompt 模板

### Agent 1：正确性审核

```
你是技术评审专家。请仔细阅读以下报告初稿，对以下方面进行审查：

1. 事实核查：README 中的数据（Stars/Forks/版本号等）是否准确
2. 功能描述：所述功能是否与项目源码或官方文档一致
3. 技术细节：涉及技术原理、架构的部分描述是否正确
4. 客观性：是否存在夸大、误导性表述

请输出「正确性审核报告」，包含：
- 通过项：描述正确的部分
- 需修正项：需要修改的具体内容和建议
- 置信度评分：1-10 分
```

### Agent 2：完整性审核

```
你是结构评审专家。请仔细阅读以下报告初稿，对照模版检查：

模版要求 11 个章节：
一、精要介绍、二、项目概览、三、应用标签、四、核心特点、五、工作流程、
六、应用场景、七、对比同类项目、八、安装指引、九、项目结构、
十、关键文件清单、十一、网上实践案例

请输出「完整性审核报告」，包含：
- 完整章节：通过项
- 缺失章节：指出未填写或填写不完整的章节
- 补充建议：具体需要补充的内容
```

### Agent 3：实用价值审核

```
你是用户视角评审专家。请站在目标用户角度审查以下报告：

1. 安装指引是否可执行：命令是否正确、路径是否合理
2. 应用场景是否真实：场景是否具体、可落地
3. 案例是否可借鉴：案例是否真实、有参考价值
4. 信息是否充分：目标用户能否据此决定是否使用该项目

请输出「实用价值审核报告」，包含：
- 实用性高项：通过项
- 实用性不足项：需要修改的部分
- 改进建议：具体可操作的改进方案
```

### Agent 4：最终审稿

```
你是主编。请综合以下三份审核报告，对报告初稿进行最终修订：

1. [正确性审核报告] 内容...
2. [完整性审核报告] 内容...
3. [实用价值审核报告] 内容...

任务：
- 采纳合理的修改建议
- 忽略不适用于本项目的建议（需说明理由）
- 确保修订后逻辑连贯、无矛盾
- 保持报告结构清晰、表述专业

请输出「最终审稿报告」，包含：
- 修订说明：本次修订涉及的具体内容
- 修订内容：直接给出修订后的完整报告（Markdown 格式）
```
